Verification & Testing

Durable ~25 MIN · VIDEO + READING + EXERCISE
Learning objectives

By the end of this module you can:

  • Translate acceptance criteria into named test cases, one observable behavior per test.
  • Review generated tests with the same skepticism as generated code — and name the four smells of a weak test.
  • Prove a test can fail before trusting that it passes (the red-first discipline).
  • Run the test-first delegation workflow: approve failing tests, then let the agent implement against them.
  • State the merge gate and the protected-tests rule, and explain why each exists.
▶ Module 05 overview — includes a live red-first demonstration VIDEO · ~8 MIN · [RECORD SUMMER 2026]

1. "It ran once" is a demo, not evidence

Module 04 had you trace code by hand. That was the right way to learn to read — and the wrong way to verify a project, because hand-tracing checks one path, once, on the day you happened to be careful. Your project will change every week for the rest of the semester, much of it written by an agent, and every change can silently break what worked. Verification has to be something that re-runs.

That's all a test suite is: your acceptance criteria, written down in a form that executes. And in AI-assisted development it picks up a second job. The suite is the contract between you and the agent — the machine-checkable definition of done that survives between sessions with a collaborator that remembers nothing. Your spec said what done looks like; your tests are how anyone, human or agent, finds out whether we're there.

2. From criteria to tests: a straight line

If you wrote Module 02's acceptance criteria honestly — concrete input, observable result — turning them into tests is nearly mechanical. The StudyBuddy search criteria become:

Acceptance criterionTest
Typing cs 34 matches "CS 3443" filter_matchesCourseCode_caseInsensitive()
Typing algo matches "Algorithms Anonymous" filter_matchesPartialName()
Clearing the field restores the full list filter_emptyQuery_returnsAllGroups()
Zero matches shows the no-results message filter_noMatches_returnsEmptyList()

Three conventions to adopt now: one behavior per test, so a failure names the broken promise; test names that read as sentences, because the test list is your spec's table of contents; and at least one test per unhappy path your spec called out — plus one for the unhappy path it didn't, which is where Module 04 taught you generated code is weakest.

3. Generated tests are code you didn't write

Here's the recursion you've probably already spotted: the fastest way to get this suite is to ask the agent — and a generated test is generated code, owed the same review as everything in Module 04. Worse, weak tests are more dangerous than weak code, because a weak test certifies weak code. Four smells to check every generated test for:

  • The tautology. A test that cannot fail: asserting a value against itself, testing that the mock returns what the mock was told to return, or the literal assertNotNull on something that's always constructed. It pads the count and verifies nothing.
  • Happy-path bias. Six tests, all variations of the same successful case, none touching empty, null, zero-match, or tied inputs. Count your unhappy-path tests; if it's zero, the suite is decorative.
  • Asserting the implementation. A test that checks how ("the method calls toLowerCase()") instead of what ("the match is case-insensitive"). These tests break when the code is improved and pass when it's wrong in a new way — exactly backwards.
  • The echo. The subtlest one: an agent asked to write tests for buggy code will often write tests that enshrine the bug, because it derives expectations from the implementation in front of it. The defense is to derive tests from the spec, not the code — which is why the workflow in the next section puts tests first.
The red-first discipline

A test you've never seen fail proves nothing — it might be a tautology, an echo, or simply not running. So: for each test that matters, make it fail once. Comment out the fix, flip a comparison, return the wrong value — any one-line sabotage — run the suite, watch it go red, undo. Thirty seconds per test, and now green means something.

4. The workflow: tests first, then implementation

Putting the pieces in order gives you the strongest delegation pattern in this course:

  1. Delegate the tests first. Give the agent your spec's acceptance criteria — not your implementation — and ask for failing tests, one per criterion.
  2. Review the tests against the four smells. This is the highest-leverage review you'll do: ten minutes here governs everything built against them.
  3. Red-check the ones that matter.
  4. Then delegate the implementation, with the suite as the finish line: "make these pass without modifying them."
  5. Verify green yourself. Run the suite on your machine. The agent saying tests pass is a claim; your terminal is evidence — the same hierarchy as Module 04.

Two standing rules now join your conventions file:

  • The merge gate: nothing merges with a red suite, and nothing merges that added behavior without adding a test. This is checklist item 9, joining Module 04's eight.
  • Tests are protected files. The agent never edits an existing test to make it pass — that's Module 03's goal-corruption signal, now codified. A test change is its own task, with its own justification, reviewed on its own.

5. What the suite can't see

Honesty about limits keeps the tool sharp. Your unit suite won't catch a JavaFX layout that's wrong on a small window, a confusing label, a three-second freeze on a large file, or the security problems Module 06 covers — and 100% coverage of code you don't understand is still code you don't understand. Keep a short manual checklist for the UI behaviors your spec promised, run it before each demo, and treat the suite as necessary, never sufficient. Verification is a portfolio; tests are its largest holding.

6. Hands-on exercise (~20 min)

Work against the feature you merged in Module 04.

  1. Delegate test generation from your Module 02 acceptance criteria — paste the criteria, not the code, and ask for one named test per criterion plus one unhappy path of the agent's choosing.
  2. Review against the four smells. Find at least one weakness — there almost always is one — and strengthen it yourself: a real assertion, a nastier input, a behavior instead of an implementation detail.
  3. Red-check two tests by sabotaging the code, watching the failure, and reverting. Save the red output.
  4. Run the full suite yourself and merge through the gate: green suite, review note updated with what you found in the tests.
Bring to class

Your weakest generated test (before and after you strengthened it) and your red-check output. The studio session plays "can this test fail?" — the room inspects volunteered tests and votes before the sabotage run. Being confidently wrong here, in public, for free, is the cheapest testing lesson you'll ever get.

7. Going deeper

  • Your testing framework's docs on parameterized tests — the four search criteria above collapse into one table-driven test, and agents write these well when asked.
  • The classic test-smell catalogs (start with "test smells" in the software-engineering literature) — written for human-authored tests, newly relevant for generated ones.
