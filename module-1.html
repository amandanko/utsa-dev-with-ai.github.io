The Landscape & Your Stance

Durable ~20 MIN · VIDEO + READING + EXERCISE
Learning objectives

By the end of this module you can:

  • Distinguish the three working modes of AI coding tools — inline assist, supervised agent, autonomous agent — and pick the right one for a given task.
  • Explain, with a concrete example, why "vibe coding" produces software you cannot defend.
  • State the course stance — you own every line you submit — and apply the CS 3443 disclosure policy to your own work.
▶ Module 01 overview VIDEO · ~6 MIN · [RECORD SUMMER 2026]

1. Three working modes, not one tool

When people say "AI coding tool," they're describing at least three different working relationships. The product names will keep changing; the modes won't.

ModeWhat it doesYou are the…Best for
Inline assist Suggests the next lines as you type; answers questions in a chat panel. Author Code you understand well and could write yourself, faster.
Supervised agent Plans and executes a multi-file task — edits, runs tests, iterates — while you watch and approve. Reviewer & director Well-specified features, refactors, and test scaffolding.
Autonomous agent Works in the background from a task description and returns a finished change. Client & auditor Routine, low-risk, easily verified work. Used sparingly in this course.

The skill this course builds is not "prompting." It is choosing the right mode for the task in front of you, and supplying what each mode needs from you: clear specification going in, and rigorous review coming out. The further right you move in that table, the less code you type — and the more your judgment matters.

2. What vibe coding actually costs

Vibe coding means prompting a model to generate software you don't read or understand, judging it only by whether it appears to run. It's a fine way to make a throwaway demo. It's a terrible way to build anything you must stand behind, for three reasons you'll see repeatedly this semester:

  • You can't debug what you can't read. The first nontrivial failure leaves you with a codebase that is, to you, someone else's — except there's no one to ask.
  • Generated code carries real defect and security rates. Industry reviews in 2026 still find security flaws in a large fraction of AI-generated code, which is why professional teams require human review before merging. Module 06 covers the specific failure modes — including package hallucination, where a model confidently imports a dependency that doesn't exist, a vulnerability class documented by UTSA researchers.
  • It shortchanges the only thing you're here to build. The degree certifies your judgment, not your typing. An interviewer, a teammate, and eventually an incident review will all ask the same question: why is this code the way it is?

3. The course stance: you own every line

In CS 3443 you are encouraged to use AI tools, in any of the three modes, on your project work. One rule governs all of it:

If you submit it, you own it. You should be able to explain what any line does, why it's there, and what would break if it changed.

"The AI wrote that part" is not an explanation a professional gets to give, and it isn't one here either. In practice the stance means three habits, which Modules 04 and 05 turn into concrete technique:

  1. Read before you accept. Generated code enters your project through review, never through reflexive approval.
  2. Verify before you merge. Behavior is demonstrated by tests you understand, not by "it ran once."
  3. Disclose your process. Each project submission includes a short AI-use note: which tools, which mode, for what, and what you changed after review. Honest disclosure is never penalized; it's the professional norm you're practicing.
Academic integrity

The disclosure note is part of the assignment. Submitting AI-generated work without disclosure, or being unable to explain your own submission, is treated under the university's academic integrity policy. When unsure whether a use is allowed, ask — before the deadline, not after.

4. Hands-on exercise (~10 min)

Pick any AI tool you already have access to (a chat model is fine — no setup required for this one; playbooks come into play in Module 03).

  1. Ask it to generate a small, complete program: a Java class that parses a CSV file of students and returns the top three by GPA, with ties broken alphabetically.
  2. Without running it, read the result and write down: one thing it did well, one thing you'd change, and one question you can't answer about it.
  3. Now run it on an edge case the spec implies but doesn't state — an empty file, or two students tied on both GPA and name. Note what happens.
Bring to class

Your three written observations and the edge-case result. The Q&A session opens with a comparison: same prompt, different tools, different failures — and what that tells us about where the human fits.
