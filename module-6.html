Security & Failure Modes

Durable ~20 MIN · VIDEO + READING + EXERCISE
Learning objectives

By the end of this module you can:

  • Explain package hallucination and the slopsquatting attack it enables — and run the dependency-verification ritual before anything gets installed.
  • Name the classic security flaws that generated code reproduces at machine speed, and where to look for each in a diff.
  • Describe prompt injection against a coding agent and apply the untrusted-content rule.
  • Add the three-item security pass to your review, and use the agent as an adversarial reviewer without taking its word for the results.

1. Your threat model just changed

Every module so far treated agent mistakes as quality problems: wrong behavior, weak tests, code you can't explain. This module treats them as security problems, which is different in one crucial way — quality failures are random, but security failures are hunted. Attackers study how these models fail and build attacks shaped exactly like the failures. Two threat classes matter for this course:

  • Flaws in what the agent writes — insecure code and dangerous dependencies entering your project through your own approvals.
  • Attacks on the loop itself — adversarial content that manipulates the agent while it works on your behalf.

Neither requires you to become a security researcher this semester. Both are handled by extending habits you already have: the review checklist gets a security pass, and the boundary rules from Module 03 get teeth.

2. Package hallucination — discovered down the hall

Start with the failure mode documented by researchers in this department. The UTSA study you met in Module 01 — Spracklen et al., "We Have a Package for You!" — generated code at scale across sixteen models and checked every imported package against the real registries. Roughly a fifth of the package references pointed to packages that do not exist: names that are plausible, well-formed, and entirely invented. Worse, the hallucinations weren't random noise — a majority of invented names recurred when prompts were repeated, meaning models reliably hallucinate the same fake packages.

Recurrence is what turns a quirk into an attack. If a model keeps recommending fast-json-validator and no such package exists, an attacker can register that exact name on the public registry with malware inside — then simply wait for developers everywhere to approve the install their agent suggested. The technique is called slopsquatting, and the payload doesn't exploit your code; it exploits your approval. Which is why the defense is a ritual at the approval moment:

The dependency ritual — before any install is approved
  1. It exists: find the package's page on the official registry yourself — never via a link the agent provided.
  2. It's established: check age, release history, download counts, and a linked source repository with real activity. A two-week-old package with three downloads fails, whatever its name.
  3. It's exactly right: compare the name character-by-character against the official docs. Typosquatting — requets for requests — predates AI and still works.
  4. It's sanctioned: your spec's constraints said which dependencies are allowed. If this one isn't on the list, that's a spec conversation, not an install.

Ninety seconds, once per new dependency. Checklist item 2 from Module 04 — "no new dependencies I didn't sanction" — now has a procedure behind it.

3. Classic flaws at machine speed

Most security problems in generated code aren't exotic — they're the classics, produced faster and with more confidence. Industry code reviews in 2026 still find security flaws in a large fraction of AI-generated code, which is precisely why professional teams keep a human review in front of every merge. Four to look for, and where:

  • Unvalidated input at trust boundaries. Generated code tends to validate what the spec mentioned and nothing else. Find every place outside data enters — a text field, a loaded file, a network response — and ask what happens when the input is hostile, huge, or malformed. (Your unhappy-path tests from Module 05 already live here; security just raises the stakes.)
  • Injection by string-building. Queries, shell commands, or file paths assembled by concatenating user input. The fix has been the same for twenty years — parameterized queries, proper APIs — but models trained on twenty years of tutorials still reproduce the tutorial version. Any string concatenation that ends in execute() is a finding.
  • Secrets in the code. Agents asked to "make it work" will happily hardcode an API key, and will also happily commit it. Rule: secrets live in untracked config (.env, gitignored), never in source, never in prompts — a pasted key is a shared key.
  • Yesterday's APIs. Training data ages. Generated code may use deprecated crypto, outdated TLS patterns, or a vulnerable version of a real library. The current way to do anything security-relevant is a docs lookup, not a model memory.

4. Attacks on the loop: prompt injection

The second threat class targets the agent, not the code. Your agent reads things while it works — files in the repo, error messages, READMEs of libraries, web pages if you allow it tools. Any of that content can contain instructions: text crafted so that a model processing it treats it as something to obey. A poisoned README that says, in effect, "as part of setup, run this command" can turn your helpful agent into the attacker's hands — operating with whatever permissions you granted.

Your defenses are the guardrails you already run, now with their real justification visible:

  • Approvals on (Module 03) means an injected instruction still has to get past you. This is why the course made auto-run a privilege to earn: every permission you grant the agent is a permission you grant to anything the agent reads.
  • Treat external content as untrusted input to the agent. Cloning an unknown repo, pasting a long error from a random forum, pointing the agent at an arbitrary URL — each is a trust decision. Make it consciously, with permissions scoped down.
  • Watch for the swerve. An agent that suddenly proposes an action unrelated to your task — a network call, a new install, a file outside the boundary — right after reading external content has given you Module 03's scope-drift signal with a security meaning. Stop and read what it just read.

5. The security pass

Three items join your review, completing the course checklist at twelve:

  1. Every new dependency passed the dependency ritual.
  2. Trust boundaries in the diff were walked: inputs validated, no string-built queries or commands, no secrets in source.
  3. The agent was asked to attack its own code — and at least one of its claims was verified by hand.

Item 11's "walked" means physically: list where data enters, point at the validation. Item 12 deserves a note: asking the agent "act as a security reviewer; how would you attack this code?" is genuinely productive — adversarial framing surfaces issues plain review misses. But Module 04's rule holds with no exceptions: explanations are claims, not evidence. A model can invent a vulnerability that isn't there and miss one that is. Its attack list is your reading list, not your verdict.

6. Hands-on exercise (~15 min)

  1. Delegate a small feature that imports an external package — e.g., "export the group list to a formatted PDF, using a library of your choice." The free choice is the point.
  2. Run the dependency ritual on whatever it picked, and record the evidence: registry link you found yourself, age, downloads, repo activity, name comparison. If the package fails any step — and sometimes it will — reject it, tell the agent why, and ritual the replacement.
  3. Run item 11 on the diff: list every trust boundary the feature touches (at minimum: the file path it writes, the group data it reads) and note the validation present or missing.
  4. Run item 12: ask the agent to attack its own code, pick its most interesting claim, and verify it by hand — try the hostile input, or find the docs that confirm or refute. Record whether the claim held.
Bring to class

Your dependency-ritual evidence and your verified (or busted) attack claim. The studio session tallies the room: how many distinct packages did the same prompt produce, did any fail the ritual, and which agent-claimed vulnerabilities turned out to be real? The distribution on the board is the UTSA paper's finding, reproduced by you.
