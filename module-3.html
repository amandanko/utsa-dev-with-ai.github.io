Working with an Agent

Durable ~25 MIN · VIDEO + READING + EXERCISE
Learning objectives

By the end of this module you can:

  • Configure an agent with the course safety settings — approvals on, auto-run off — and use version control as your undo button.
  • Scope a task to agent size: specified, bounded, and verifiable, producing one reviewable diff.
  • Assemble the context an agent needs — and recognize that it knows nothing you didn't put in front of it.
  • Run the delegate → watch → review → iterate loop, and name the four signals that mean it's time to take the wheel.

Before you start: this module is hands-on. Pick a tool from the playbooks and complete its setup section first (~10 minutes, not counted above).

1. From chat to agent: what actually changes

A chat model gives you text; you decide what to do with it. An agent acts: it reads your files, edits them, runs your build, executes tests, and iterates on failures — in your project, with real consequences. That changes your job description. In chat you were the author copying answers; with an agent you are a director and reviewer, and the quality of your direction (Module 02's spec) and your review (Modules 04–05) are now the entire value you add.

It also changes the failure stakes. A bad chat answer wastes your time. A badly supervised agent can overwrite working code, install a dependency you didn't sanction, or "fix" a test by deleting it. None of these are reasons to avoid agents; all of them are reasons the next section exists.

2. Set up for safety first

Two non-negotiable course rules, whatever tool you choose:

  1. Approvals on, auto-run off. The agent proposes; you approve each file edit and each command, at least until Module 05's verification habits are solid. Every playbook's setup section shows where this switch lives. An agent that can run anything without asking is the autonomous mode from Module 01 — earn it before you use it.
  2. Commit before you delegate. Version control is your undo button. Start every agent session from a clean commit, so the agent's entire contribution is one inspectable diff — and one git reset away from gone. If you're about to delegate and git status isn't clean, stop.
Sharp edge

Agents are eager. Given a vague task and broad permissions, a typical agent will touch more files than you expected and call it initiative. The guardrails above don't slow you down; they make the agent's eagerness reviewable instead of dangerous.

3. Scoping: what makes a good agent task

The unit of delegation is not "the feature." It's one reviewable diff — a change small enough that you can read all of it, understand all of it, and check it against acceptance criteria in one sitting. Three tests for a well-scoped task:

  • Specified: it has the five-part spec from Module 02, or it's small enough that the spec fits in two sentences without ambiguity.
  • Bounded: you can name the files it should touch — and you say so, which gives your review a tripwire: any file outside the list needs a justification.
  • Verifiable: you know, before delegating, how you'll check the result. If you can't say what done looks like, the agent can't either.

Big features get decomposed first. The StudyBuddy search feature from Module 02 is one diff. "Build the whole StudyBuddy app" is not a task; it's a semester. Decomposition — slicing a feature into ordered, individually verifiable steps — is itself design work you can co-do with the agent: ask it to propose a plan, then edit the plan before any code happens. Approving a plan is cheaper than rejecting a diff.

4. Context: the agent knows only what's in front of it

An agent reading your repository is not the same as an agent understanding your project. It doesn't know your team's conventions, your professor's requirements, or the architectural decision you made last week in a Discord thread. Feed it deliberately:

  • The spec — verbatim, not paraphrased from memory.
  • Pointers, not dumps — name the two or three files that matter ("the list view is GroupListView.java") rather than pasting the whole project. Modern agents can search your repo; directing their attention is faster and cheaper than flooding it.
  • A conventions file — most agents read a project-notes file from your repo root (each playbook names the current convention). Your team writes it once: language version, architecture, build and test commands, style rules, things never to touch. This file is your team's standing spec, and keeping it current is a graded part of the project.

A useful mental model: every session, you're onboarding a skilled contractor with no memory of yesterday. What would they need to read first? That's your context.

5. The loop — and when to take the wheel

The working rhythm is the same in every tool: delegate → watch → review → iterate. Delegate one scoped task. Watch the plan and the commands — approving is reading, not clicking. Review the diff against your acceptance criteria (Module 04 makes this rigorous). Then iterate: feedback to the agent is spec refinement, specific and observable, not "that's wrong, try again."

And know the exits. Four signals that the supervised loop has stopped working and you should take the wheel — fix it yourself, or reset and re-scope:

  1. Thrashing. The same error "fixed" three times, each fix creating the next error. The agent is guessing; more iterations buy more guesses.
  2. Scope drift. Files outside your named boundary are changing, or the diff keeps growing past what you can review.
  3. Confident nonsense. It cites an API or method you can't find in the docs, or a package you can't find in the registry — the hallucination pattern Module 06 examines in depth. Verify; never assume your ignorance.
  4. Goal corruption. The acceptance criterion was "tests pass" and the agent is editing the tests. The letter of the spec is beating its spirit; stop and restate what done means.

Taking the wheel isn't failure — it's the judgment this course certifies. The professional skill is noticing in two minutes, not twenty.

6. Choose your team's tool (this week)

Your team commits to a primary tool for the project this week. Use the decision guide playbook; weigh access and cost on student plans, which modes it supports, how good its review surface is (can you see every diff and command easily?), and whether it reads images — which your Module 02 sketches will reward. Disagreement inside the team is fine; the conventions file and the review workflow are what you must share.

7. Hands-on exercise (~20 min)

Use your chosen tool on the course sample project (or your team repo if it's set up).

  1. Confirm the safety settings from your tool's playbook, and commit so git status is clean.
  2. Write a two-line conventions file if the project doesn't have one (build command, test command, "don't modify the data layer").
  3. Delegate your Module 02 spec — the one you already wrote. Ask for a plan first; edit the plan if it touches files outside your boundary.
  4. Watch the run. Approve deliberately. Note every moment you intervened — or realize in hindsight that you should have.
  5. Review the final diff against your acceptance criteria. Record: diff size in lines, files touched vs. files you named, criteria met, and one thing the agent did that you didn't ask for.
Bring to class

Your run notes and the diff. The studio session collects the room's "moment I should have taken the wheel" stories and maps them onto the four signals — by the end, you'll have seen a dozen failure shapes you haven't personally hit yet.

8. Going deeper

  • Your tool's playbook, sections 4–5 — the review workflow and the current sharp edges your classmates have logged.
  • Your tool's official docs on permissions and project-notes conventions — names change quarterly; the concepts here don't.
  • Module 04 preview: the diff you just produced is the reading material. Don't merge it yet.
