Tool Playbooks / Claude Code

Getting started with an agentic CLI: Claude Code

Verified · Jun 2026 TERMINAL-NATIVE AGENT · ANTHROPIC
Shelf-life warning

This is tool-layer content. Commands, settings names, and prices below were checked against the official docs on the verified-by date and drift quickly. When something doesn't match what you see, trust your screen, check the official docs, and open a PR — Module 08 shows you how.

1. What it is & which mode

Claude Code is Anthropic's coding agent. Its native home is the terminal — you run claude inside a project directory and converse there — with optional VS Code and JetBrains extensions that render its diffs in your IDE, and a desktop app if you'd rather avoid the terminal entirely.

In Module 01's taxonomy it is a supervised agent as this course configures it: it plans multi-step tasks, edits files, and runs build and test commands, asking your approval at each consequential step. It is capable of running far more autonomously; those modes stay off in this course (see the settings below), and that's a feature — the skills you're practicing are direction and review.

2. Setup for this course

Access first. Claude Code requires a paid plan (Claude Pro or above) or pay-per-token API billing — there is no free tier as of the verified-by date. Check the course syllabus or ask your instructor before paying for anything: course-provided access, education programs, and pricing all change frequently, and your team only needs the tool on the machines that will run delegation sessions.

Install. The native installer is the recommended method (no Node.js required; auto-updates itself):

  • macOS / Linux / WSL: curl -fsSL https://claude.ai/install.sh | bash
  • Windows PowerShell: irm https://claude.ai/install.ps1 | iex

Open a fresh terminal, confirm with claude --version, then run claude once from any directory — the first launch opens your browser to sign in.

Course-required safety settings. Good news: the defaults are the course rules. Out of the box, Claude Code asks before every file edit and every shell command. Your job is to not loosen that yet:

  • Run /permissions inside a session to see the current rules and which settings file each comes from. Rules evaluate in deny → ask → allow order and live in .claude/settings.json (project, committed — so your team shares one configuration) or settings.local.json (personal, gitignored).
  • It's fine to allow read-only and verification commands your team runs constantly — git status, git diff, mvn test. Keep edits, installs, git push, and anything destructive on ask.
  • Never use --dangerously-skip-permissions in this course. The flag name is honest. An agent that can run anything without asking is Module 01's autonomous mode plus Module 06's prompt-injection surface, on your machine, with your files.

3. Your first task

Work in the course sample project (Module 03's exercise):

  1. cd studybuddy and confirm git status is clean — the Module 03 rule doesn't care which tool you're holding.
  2. Create the conventions file Claude Code reads automatically: a CLAUDE.md at the repo root. Ours can be two lines — an import of the project notes, plus the commands:
    @PROJECT_NOTES.md
    Build: mvn javafx:run · Tests: mvn test
  3. Start claude, then ask for a plan before any code — paste your Module 02 spec and add: "Propose a plan first; don't edit anything yet. I expect changes only in GroupListView.java."
  4. Read the plan like it's a diff (it is one, in the future tense). Edit it if it strays outside your boundary, then approve the steps one at a time.

4. Review workflow

  • During the run: every proposed file edit appears as a diff in an approval prompt — approving is reading. Every shell command shows before it runs. Press Esc at any moment to interrupt; the agent stops and waits.
  • Checkpoints: Claude Code snapshots the code before each of its edits. Double-tap Esc or use /rewind to step back, restoring the code, the conversation, or both. Know the limit: checkpoints cover Claude's edits only — not your edits, and not the effects of shell commands. Git remains the real undo button, which is why you committed clean in step 1.
  • After the run: git diff is your one reviewable diff for the Module 04 three-pass review. The session transcript scrolls back through every command that ran — your evidence for checklist item 1 (files touched) and the security pass.
  • IDE option: the VS Code extension shows the same diffs in the editor's native diff viewer and adds a per-message rewind button — same workflow, friendlier rendering.

5. Known sharp edges

  • Cost and usage limits. Paid plans meter usage in time-windowed limits; a long thrashing session can hit them mid-task. The course workflow is the mitigation: small scoped tasks, and Module 07's two-attempt ladder before any marathon.
  • Eagerness. Given a vague task it will plan generously — more files, more "improvements." Plan-first prompting and a named file boundary (step 3 above) keep it honest. It may also offer to add frequently-approved commands to your allow list; review those suggestions like any diff.
  • Long sessions compact themselves. Very long conversations get summarized in place to fit context, and details (including your careful spec) can blur. One task per session; start fresh rather than continuing a marathon — your spec and CLAUDE.md are the durable memory, not the chat scroll.
  • CLAUDE.md is advice, not law. Instructions in CLAUDE.md shape what the agent tries to do; only permission rules change what it's allowed to do. Conventions go in CLAUDE.md; protections go in .claude/settings.json.
