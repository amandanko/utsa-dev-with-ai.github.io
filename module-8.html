Keeping Up

Durable ~20 MIN · VIDEO + READING + CAPSTONE EXERCISE
Learning objectives

By the end of this module you can:

  • Run the 30-minute evaluation protocol on a tool you've never seen — using artifacts you already own as the benchmark.
  • Read a tool announcement critically: separate a new capability from a renamed one, and treat benchmark numbers as claims awaiting evidence.
  • Author a playbook in the six-section template and contribute it by pull request.
  • Describe a sustainable staying-current practice — a cadence, not a firehose.

1. The half-life problem — and this site's answer to it

Time to say the quiet part out loud. Every tool playbook on this site carries a verified date because we expect it to be wrong within months — renamed features, changed pricing, a whole product rebranded over a weekend. That's not sloppiness in the tool layer; it's the truth about the territory. Meanwhile, nothing in Modules 01–07 has a date on it, because specification, review, verification, security posture, and debugging discipline don't expire when a vendor ships.

You've been living inside this architecture all semester. Now it's yours to carry: between this course and your graduation, the tool landscape will turn over more than once, and it will keep turning for your entire career. The professionals who handle that well aren't the ones who chase every release — they're the ones who can evaluate anything new, quickly, against standards they already hold. This module makes that a procedure.

2. What changes, what doesn't

Changes constantlyHasn't changed all semester
Product names, pricing, plansA spec decides; silence delegates the decision
UIs, settings locations, file conventionsOne reviewable diff; approve by reading
Context sizes, model versions, benchmark scoresTests are the contract; red before green
Autonomy levels and agent "modes"Every dependency gets the ritual
Which tool is "best" this quarterExplanations are claims, not evidence

Two habits for reading the announcement firehose with that table in mind:

  • The renaming trap. Marketing reintroduces existing capabilities under new names constantly. Before treating a feature as new, map it onto Module 01's three modes: is this a genuinely new working relationship, or a supervised agent with a fresh coat of paint? Most announcements are the second. The ones that are the first — a real shift in what you can delegate or how you verify it — are the ones worth thirty minutes.
  • Benchmarks are claims. A leaderboard score is a vendor-selected measurement of performance on someone else's tasks. It's not meaningless — but the claims-not-evidence rule makes its final appearance here: the only benchmark that settles whether a tool helps you is the tool running your task. Which is exactly what the protocol below does.

3. The 30-minute evaluation protocol

Here's the quiet payoff of the semester: you already own a personal benchmark suite. A spec you know intimately (Module 02's), acceptance criteria with tests (Module 05's), and a twelve-item review checklist. Evaluating a new tool means running the tool through them:

  1. Before installing (5 min). Who makes it, what does it cost on a student/free tier, and — first question always — what is its permissions model? If the docs can't tell you how to require approvals and disable auto-run, the evaluation is over and the verdict is no.
  2. Setup with safety on (5 min). Install, find the approval controls, point it at a scratch clone of the sample project. Clean git status before anything else — the Module 03 rules don't pause for novelty.
  3. The standard task (10 min). Give it your known spec, verbatim — the same one every tool you've used got. Because you know this task cold, every difference you observe is signal about the tool, not noise about the task. Watch how it plans, what it asks, what it touches.
  4. Review the diff (5 min). Twelve items, same as always. Run your existing tests against its output. A tool whose output fails your suite has measured itself.
  5. One deliberate probe (5 min). Give it something slightly vague and watch the failure shape: does it ask a clarifying question, state its assumptions, or silently sprawl across files it wasn't given? How a tool behaves at the edge of its instructions tells you more than how it behaves in the center.

Then write the verdict in one honest paragraph: what mode it really is, what it did better or worse than your current tool, the sharpest edge you hit, and whether it earns a place in your kit. Thirty minutes, repeatable forever, and notice what it actually is: a tiny version of the methodology in the UTSA package-hallucination paper — a fixed procedure, run across tools, with the differences recorded. Evaluation is just research at personal scale.

4. Write it down: the playbook

An evaluation that lives in your head dies there. The six-section playbook template is the protocol's output format: what it is and which mode, setup for this course, a first task, the review workflow, known sharp edges, and a verified-by line with your name on it. Drafting one takes about as long as the evaluation did, and it's this module's exercise — the strongest drafts are merged into this site by pull request each semester, credited, where they'll onboard students you'll never meet until someone re-runs the protocol and replaces them. That's not a side effect of how this site stays alive. It's the entire mechanism.

5. A practice, not a firehose

Finally, calibrate the ongoing effort, because "keeping up" fails in two directions: ignoring the field until your skills strand, or drowning in release notes at the cost of actually building things. The sustainable middle is a cadence:

  • Re-evaluate quarterly, not weekly. Run the protocol when something claims a new mode — a different working relationship — not a new version number. A few times a year is plenty; the protocol makes each one cheap.
  • Pick one or two high-signal sources — an official changelog, one practitioner whose evaluations you've learned to trust — over ten feeds of launch-day enthusiasm. Skepticism is cheaper at the source.
  • Keep the conventions file alive. On any team you join, that file (or its equivalent) is where tool judgment compounds. The intern who updates it thoughtfully is doing senior work.

6. Capstone exercise (~45 min)

  1. Pick a tool you have never used from the instructor's candidate list (kept current each semester — tools with student- accessible tiers, including at least one this site has no playbook for).
  2. Run the full protocol, keeping notes at each step. Time-box honestly; running over budget is itself a finding about the tool's onboarding.
  3. Draft a playbook in the six-section template — or, if the site already covers your tool, a dated update PR correcting what's drifted since its verified-by line.
  4. End with the one-paragraph verdict, including the sentence: "Compared to my team's current tool, I would / would not switch, because…"
Bring to class

Your verdict and your draft. The final studio session is lightning rounds — two minutes per tool — and then the room compares verdicts where two people evaluated the same tool. When trained evaluators disagree, the discussion about why is the last and best lesson of the semester. Strong drafts get merged, with credit, before finals.

7. Where this leaves you

A closing inventory, because you've earned one. You can turn an idea into a spec that decides things; delegate it as one reviewable diff to an agent you've configured to be safe; read what comes back in three passes and explain every line; hold it to tests that can fail; refuse a dependency that can't prove it exists; debug without thrashing when it breaks at 11 p.m.; and — as of this module — do all of it again with whatever tool exists the year you're reading this. Take the checklist, the protocol, and the habit of writing things down into your internship. The tools in this site's playbooks will be museum pieces before you're a senior engineer. What you do with them won't.

  • Re-read the Spracklen et al. methodology section with new eyes — you've now run the personal-scale version of it.
  • Star the course repository. When a playbook you wrote goes stale and a future student's PR replaces it, you'll get the notification — and that's the system working.
