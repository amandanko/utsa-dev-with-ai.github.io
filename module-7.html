Modules / 07

Troubleshooting & Debugging with AI

Durable ~20 MIN · VIDEO + READING + EXERCISE
Learning objectives

By the end of this module you can:

  • Write a bug report that makes an agent useful: expected vs. actual, minimal reproduction, exact evidence, and what's already ruled out.
  • Use the agent for hypothesis generation while keeping hypothesis testing for yourself — and refuse fixes for hypotheses nobody confirmed.
  • Run the two-attempt escalation ladder, and climb off it into classical debugging before the thrash loop starts.
  • Debug in the right blame order when the code was generated: spec first, code second, environment third, tool last.

1. Debugging didn't go away — it moved

It's 11 p.m., the demo is tomorrow, and the feature that worked on Tuesday throws an exception nobody recognizes. This is the moment the whole course has been preparing you for, because it's the moment vibe coding presents its bill: a codebase you can't read is a codebase you can't fix. If you've been running the Module 04–05 habits, you arrive with assets instead — a suite that tells you what broke, review notes that tell you what each piece was for, and a commit history where every agent contribution is one clean diff.

The temptation at 11 p.m. has a specific shape: paste the stack trace, type "fix it," accept the patch, repeat. Sometimes it even works. But it's a slot machine — each pull costs a diff you didn't really review, and three pulls in you have new code layered on broken code and no idea which change did what. Module 03 named this thrashing and told you to exit. This module is the full discipline of staying out of it.

2. The bug report is a spec

Everything Module 02 taught about specification applies double here, because a debugging agent with a vague prompt doesn't just guess — it guesses confidently, with a patch attached. A useful bug report has four parts, and they're worth two minutes even at 11 p.m.:

  • Expected vs. actual — and "expected" comes from your acceptance criteria, verbatim. "Typing cs 34 should match CS 3443; instead the list goes empty."
  • Minimal reproduction — the smallest input and shortest path that triggers it. Shrinking the repro is half the diagnosis, and it's work you can co-do with the agent: "help me find the smallest input that still fails."
  • Exact evidence — the complete error text and stack trace, pasted, not paraphrased. Models read stack traces well; they read "it crashes somewhere in the filter thing" the way you'd expect.
  • What's ruled out — "the suite passed before commit a3f1; the repository returns the right list (verified by test); the bug appears only when the query contains a space." Every ruled-out fact prunes the agent's hypothesis tree before it grows.

3. Hypotheses are the agent's job; verification is yours

Here's the division of labor that makes AI genuinely powerful at debugging. Models are exceptional at differential diagnosis — given a good report, they'll produce a ranked list of plausible causes in seconds, including ones outside your experience. So ask for exactly that: "List the most likely causes, ranked, with how I could cheaply test each one — don't write a fix yet."

That last clause is the discipline. The failure pattern to avoid is accepting a patch for a hypothesis nobody confirmed. A fix that makes the symptom disappear without a confirmed cause is the worst outcome in debugging: the fix that works for the wrong reason — the real bug is still there, now wearing camouflage, and it will be back the week of the final. The loop you want instead:

  1. Agent proposes ranked hypotheses with a cheap discriminating test for each.
  2. You run the cheapest test — a print statement, a breakpoint, one assertion — and observe.
  3. Confirmed? Now request the fix, for that cause, and review it like any diff (it's amazing how small fixes get when the cause is known). Refuted? Feed the observation back; the new fact is worth ten retries.

One mechanical rule keeps the whole thing sane: one attempted fix, one commit. Never let the agent stack a second fix on an unverified first. If two fixes fail, revert both — you are demonstrably guessing, and the ladder below says what's next.

4. The escalation ladder

Module 03 gave you the thrashing signal; here's the procedure that replaces it. Two assisted attempts, then the wheel:

  1. Attempt 1: full bug report, hypothesis loop, one verified fix. This resolves most bugs.
  2. Attempt 2: only with new evidence — a fact attempt 1 produced. "Try again" is not an attempt; it's the same pull of the same slot machine. If you have nothing new to add, you're already at step 3.
  3. Take the wheel. Close the agent panel and reach for the classical kit, which works exactly as well as it did before AI:
    • The debugger. Breakpoint at the last place you know the data is right; step until it's wrong. The IDE debugger is still the highest-bandwidth instrument in this trade, and CS 3443 expects you fluent in it.
    • Bisection. Your one-diff-per-task git history (Module 03) makes git bisect almost magical: a few checkouts and the guilty commit identifies itself.
    • Rubber-ducking — with a twist. Explain the failing code line by line, out loud or in writing, to the agent. The explaining is what finds the bug; the agent is just a duck that occasionally asks a good question. If there's a line you can't explain, Module 04 already told you: that line is the homework, and tonight it's also the prime suspect.

The ladder isn't anti-AI — you can bring findings from step 3 back to the agent for a final fix. It's anti-thrash: a hard stop that converts "one more try" into evidence-gathering.

5. When the code was generated: blame in the right order

Debugging code an agent wrote adds one diagnostic layer, and it goes first:

  1. The spec. Reread what you actually asked for. A startling fraction of "bugs" in generated code are the model doing precisely what you said and not what you meant — the silent decisions from Module 02, surfacing late. If the spec was wrong, the fix is a spec fix; patching the code leaves you maintaining a correct implementation of the wrong thing.
  2. The code. Now hunt with the Module 04 smell list in hand — silent defaults and swallowed exceptions are disproportionately where generated bugs hide, because they're designed not to crash. A bug with no stack trace is usually a catch block eating one; your first move is often to make the code louder.
  3. The environment. Version mismatches, stale builds, the dependency that resolved differently on your teammate's machine (Module 06's territory). "Works on my machine" is an environment hypothesis, and it's testable.
  4. The tool. Last, and rarely: the playbook's sharp-edges section lists the current known quirks. Check it before deciding you've found a new one — then log it there if you truly have, because that's the tool layer doing its job.

6. Hands-on exercise (~20 min, in pairs)

Sabotage night. You'll need a teammate and your merged project feature.

  1. Your partner introduces one subtle bug into your feature branch — a flipped comparison, an off-by-one, a swallowed exception — commits it among two innocent cosmetic commits, and tells you only the symptom.
  2. Write the four-part bug report before touching the agent. Yes, all four parts.
  3. Run the ladder: hypothesis loop, one fix per commit, two-attempt limit. If you hit step 3, use the debugger or git bisect — with three commits, bisect will feel like cheating. That's the lesson.
  4. Record: your report, the hypotheses generated, the discriminating test you ran, attempts used, and whether the wheel was needed. Then swap roles.
Bring to class

Your run record and your partner's bug. The studio session runs the room's stats — attempts-to-fix, how often the spec was the real culprit, which discriminating tests paid off — and then the instructor debugs the session's nastiest surviving bug live, narrating the ladder.
