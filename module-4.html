Reading Code You Didn't Write

Durable ~25 MIN · VIDEO + READING + EXERCISE
Learning objectives

By the end of this module you can:

  • Review a generated diff with the three-pass method: intent, structure, then line-level.
  • Spot the smells characteristic of generated code — silent defaults, speculative generality, plausible-but-wrong APIs, ignored project conventions.
  • Verify what you read: check unfamiliar APIs against documentation and treat "I can't explain this line" as an action item, not a shrug.
  • Make and record a merge decision — accept, request changes, or reject — with review notes that feed your AI-use disclosure.

Before you start: have your unmerged diff from Module 03 open. It's this module's reading material.

1. Reading is the job now

In the pre-agent world, you read a little code and wrote a lot. That ratio has inverted. When an agent produces most of the lines, the engineering happens in the reading — which is why professional teams still put a human review between generated code and the main branch, and why senior engineers describe review as where they now spend their time. Review is also the only mechanism that makes Module 01's stance real: you own every line is an empty slogan unless there's a moment where you actually read every line and decide it earns a merge.

One reframe before the technique: reviewing generated code is not proofreading. Proofreading assumes the author understood what they meant and checks the surface. Your author is a system that produces plausible code without understanding your project — so plausibility is exactly the thing you cannot trust. You're not checking spelling; you're checking truth.

2. The three-pass method

Reading a diff line-by-line from the top is how reviews fail — you drown in detail before you've asked whether the change is even the right shape. Read in three passes, coarse to fine, and stop the moment a pass fails:

Pass 1 — Intent (2 minutes): is this the change I asked for?

  • Files touched vs. files you named in Module 03's boundary. Anything outside the list needs a justification before you read further.
  • Diff size vs. task size. A 40-line task that came back as 400 lines is a scope question, not a reading assignment.
  • Skim only the shape: new classes, new dependencies, deleted code. A new dependency you didn't sanction fails Pass 1 on the spot.

Pass 2 — Structure (5 minutes): does it belong in this project?

  • Does it follow your architecture and your conventions file — or did it import a pattern from someone else's codebase?
  • Did it reuse your existing helpers, or quietly reimplement them? Agents duplicate what they don't notice.
  • Are names, error handling, and style consistent with the surrounding code? Inconsistency here predicts inconsistency you can't see.

Pass 3 — Lines (the real work): is it true?

  • Trace the main path by hand against an acceptance criterion: pick a concrete input from your spec and walk it through the code on paper.
  • Trace one unhappy path: the empty list, the null field, the zero-match query. Generated code is weakest exactly where your spec was silent.
  • Interrogate every line you can't explain. That's not a vague ideal — it's the checklist item below.
The generated-code review checklist
  1. Every touched file was in scope, or has a justification I accept.
  2. No new dependencies I didn't sanction. (Module 06 raises the stakes on this one.)
  3. Existing project helpers were reused, not reimplemented.
  4. Every unfamiliar API call was checked against real documentation.
  5. I traced one happy path and one unhappy path by hand.
  6. No swallowed exceptions, silent fallbacks, or magic default values I didn't choose.
  7. No deleted or weakened tests, and no dead code along for the ride.
  8. There is no line I cannot explain to a teammate.

Print it, or copy it into your pull-request template — that's where it lives on professional teams.

3. Smells specific to generated code

Generated code has its own characteristic failure smells, different from the ones classic code-review guides catalog. Five to develop a nose for:

  • Silent defaults. A catch block that logs and continues; a missing config value replaced by a guessed constant; a parse failure that returns an empty list. The code never crashes — and never tells you it's wrong. In a diff, every catch, every ?: fallback, and every default parameter deserves a pause: who chose this behavior? If the answer isn't "my spec," it's a finding.
  • Speculative generality. An interface with one implementation, a config option nobody asked for, a "manager" class wrapping two method calls. It reads as professionalism; it's padding that you will maintain. Your non-goals section is the measuring stick.
  • Plausible-but-wrong API usage. A method that almost exists, an overload with the arguments in a believable-but-incorrect order, a deprecated call that worked in the model's training data. This is why checklist item 4 says real documentation — not the agent's word for it.
  • Convention blindness. Correct code in the wrong dialect: different naming, a second way of doing something your project already does one way. Harmless once; over a semester it turns your codebase into an anthology.
  • Decorative comments. // loop through the groups above a loop through the groups. Generated comments tend to restate the code rather than explain the why. Delete them, and add the why yourself where a future reader needs it — you're the only one who knows it.

4. Reading with the agent — without taking its word

You can and should use AI as a reading partner: ask it to summarize the diff, explain a function, or walk a specific input through the code. This is legitimately one of the best uses of these tools. One rule keeps it honest: explanations are claims, not evidence. A model can confabulate a fluent explanation of code that doesn't do what it says — including code it just wrote. So use the explanation as a map, then check the territory: run the input, read the doc, print the value. The hierarchy is docs and execution first, model claims second, your assumptions last.

And when you hit a line you can't explain even with help — an unfamiliar language feature, a clever stream pipeline, a regex — treat it as the day's homework, not an obstacle. Five minutes spent understanding one such line is the actual mechanism by which this course makes you a stronger developer than the tool. Skipping that five minutes, every time, is how vibe coding happens gradually to people who swore it wouldn't.

5. The merge decision

Every review ends in one of three recorded outcomes:

  • Accept. Checklist clean; merge it — and the commit message says what it does, not "AI changes."
  • Request changes. Findings go back to the agent as spec-grade feedback (Module 03's iterate step): specific, observable, one issue at a time.
  • Reject and take the wheel. When the findings say the approach is wrong, not the details — reset to your pre-delegation commit and either re-scope or write it yourself.

Whatever the outcome, keep two lines of review notes: what you found, and what you changed. Those notes are most of your AI-use disclosure for the assignment, written for free, at the moment you actually know the answers.

6. Hands-on exercise (~20 min)

  1. Run the three passes on your unmerged Module 03 diff, checklist in hand. Time-box Pass 1 to two minutes — the discipline is the point.
  2. Record every finding, however small: at least one scope observation, one smell (or an honest "looked for all five, found none"), and one line you initially couldn't explain, with the explanation you tracked down.
  3. Ask your tool to explain the trickiest function — then verify one specific claim in its explanation against docs or by running it. Note whether the claim held.
  4. Make the call: accept, request changes, or reject. If it's accept, merge with a real commit message. Write your two-line review note.
Bring to class

Your findings list and your merge decision. The studio session swaps diffs: you review a classmate's diff cold, with no context — which is exactly how their generated code looked to them, and the fastest way to feel why Pass 1 exists.

7. Going deeper

  • Google's engineering-practices code review guide — the standard reference for human-to-human review; notice what transfers to reviewing an agent and what doesn't.
  • Your own merged diff in a week. Re-read it; whatever confuses future-you was a finding current-you missed. Calibration is free.
