Specification Before Generation

Durable ~20 MIN · VIDEO + READING + EXERCISE
Learning objectives

By the end of this module you can:

  • Explain why specification quality, not generation speed, is the bottleneck in AI-assisted development.
  • Write an executable spec using the five-part template: context, behavior, constraints, acceptance criteria, and non-goals.
  • Use a low-fidelity prototype as a specification, and run a design conversation with an AI tool before any code exists.
  • Recognize an underspecified request by the questions a careful developer would have to ask back.
▶ Module 02 overview VIDEO · ~7 MIN · [RECORD SUMMER 2026]

1. The bottleneck moved

For most of computing history, the expensive part of software was typing it: translating an idea into correct syntax, line by line. AI agents have made that part cheap. What they have not made cheap is knowing what to build. An agent will happily produce two hundred lines from a one-sentence request — it just won't be the two hundred lines you needed, because the missing decisions didn't disappear. The agent made them for you, silently, and usually with the most generic possible answer.

This is why CS 3443 puts design before development, and why this module comes before any tool playbook. A specification is where you make the decisions that are yours to make. Everything you hand to an agent later — and everything you'll review when it comes back — is downstream of it.

A useful test: if two reasonable developers could read your request and build meaningfully different things, it isn't a spec yet. It's a wish.

2. Anatomy of an executable spec

An executable spec is a description precise enough that a competent developer — human or agent — could build from it without guessing on anything that matters. Five parts, none optional:

PartAnswersCommon failure when missing
Context What exists already? Language, framework, relevant classes, data shapes. Code that ignores your architecture — a new pattern bolted onto your project.
Behavior What happens, step by step, including the unhappy paths? Edge cases decided by the model's defaults, not by you.
Constraints What must stay true? Performance, dependencies allowed, style, APIs to use or avoid. A surprise dependency, or a rewrite of code that wasn't supposed to change.
Acceptance criteria What observable results prove it's done? "It ran once" becomes the definition of done.
Non-goals What are you deliberately not building yet? Scope creep — agents love adding features you didn't ask for.

A worked example

Suppose your team project is StudyBuddy, a JavaFX app for forming study groups. Compare two ways of requesting the same feature.

A wish

Add a search feature so users can find study groups.

Unanswered: search on what fields? Partial matches? Case sensitivity? What does the UI do with zero results? Does this filter the existing list view or open a new screen? Can it touch the data layer? An agent will answer all of these — its way.

An executable spec

Context: JavaFX app; groups live in GroupRepository.getAll() returning List<StudyGroup> (fields: name, courseCode, meetingDay); the main screen is GroupListView, which renders that list.

Behavior: Add a text field above the list. As the user types, filter the visible list to groups whose name or courseCode contains the query, case-insensitive. Empty query shows all groups. Zero matches shows the message "No groups found — try a course code like CS 3443."

Constraints: Filter in memory in the view layer; do not modify GroupRepository or add dependencies. Follow our existing MVC structure.

Acceptance criteria: typing cs 34 matches "CS 3443"; typing algo matches a group named "Algorithms Anonymous"; clearing the field restores the full list; zero-match message appears and disappears correctly.

Non-goals: no fuzzy matching, no search history, no filtering by meetingDay (planned for sprint 3).

Notice what the spec really is: a record of decisions. Writing it took ten minutes; every one of those ten minutes was design work that someone had to do. The only question is whether it's done deliberately by you, or silently by a model.

3. Prototypes are specifications too

For anything with a user interface, prose runs out of precision fast. A low-fidelity prototype — a paper sketch, a wireframe, even a labeled ASCII layout — is often the highest-bandwidth spec you can produce, and modern tools accept images directly. Two practices to build now:

  • Sketch before you prompt. Draw the screen, label the controls, number the interactions. Hand the sketch plus the five-part spec to the tool. You'll spend less time correcting layout guesses than you would have spent describing the layout in words.
  • Use the tool as a design critic before using it as a builder. Before requesting code, ask: "Here is my spec and sketch. What's ambiguous? What edge cases am I missing? Propose two alternative designs and their trade-offs." This is the co-design loop: the model is good at generating alternatives, and you are responsible for choosing. Iterate on the spec until the questions stop — then, and only then, generate.

This sequencing — sketch, spec, critique, revise, generate — is the module's core habit, and it's exactly the workflow you'll practice with a real agent in Module 03.

4. Acceptance criteria: write "done" down

Acceptance criteria deserve special attention because they do double duty. Going in, they force you to think in observable behavior ("typing cs 34 matches CS 3443") rather than vibes ("search works well"). Coming out, they are the checklist your review runs against — and in Module 05 they become tests, often generated by the same agent and verified by you. A criterion you can't imagine checking isn't a criterion; it's a hope.

Three quick quality checks for each criterion: it names a concrete input, it names an observable result, and a teammate could verify it without asking you anything.

5. Hands-on exercise (~15 min)

Use any chat-capable AI tool. You'll generate the same feature twice and compare what specification buys you.

  1. Pick a small feature from your own project idea (or use StudyBuddy's "sort groups by meeting day").
  2. Round 1 — the wish: request it in one sentence. Save the generated code without correcting anything.
  3. Round 2 — the spec: write the five-part spec (10 minutes, honestly spent), ask the tool what's ambiguous, patch the spec, then request the code.
  4. Diff the two results and write down: two decisions the model made for you in Round 1, one question the tool raised in Round 2 that you hadn't considered, and which output would be easier to review and test.
Bring to class

Both code outputs and your three written observations. The studio session pairs you up to review each other's Round 2 specs against the five-part template — the fastest way to learn what "ambiguous" looks like is to read someone else's spec cold.

6. Going deeper

  • Joel Spolsky's classic essays on painless functional specs — written decades before AI tools, and a sharper argument for this module than anything published since.
  • Your team project's design document assignment — the five-part template maps directly onto its feature sections; use it.
  • The decision guide playbook — you'll choose your team's tool in Module 03, and your spec-writing habits should inform the choice (some tools accept images and project context better than others).
